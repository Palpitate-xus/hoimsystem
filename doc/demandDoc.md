# 医院门诊信息管理系统需求文档（HIS-OP）

## 1. 项目概述

### 1.1 编写目的

本文档旨在明确医院门诊信息管理系统的功能需求、数据需求和业务规则，为系统设计、开发、测试和验收提供依据。

### 1.2 项目背景

医院门诊信息管理系统（Hospital Outpatient Information Management System，简称 HIS-OP）是现代化医院运营的必要技术支撑和基础设施。系统以病人为中心，以门诊业务流程为主线，实现挂号、就诊、收费、取药等环节的数字化管理，提高医院工作效率，改进医疗服务质量。

### 1.3 系统范围

本系统覆盖门诊业务全流程，包括：
- **门诊前台**：病人注册、挂号、预约
- **门诊医生站**：病历书写、处方开具、检查申请
- **门诊药房**：药品管理、处方发药
- **门诊收费**：费用计算、收费、退费
- **后台管理**：医生/病人/科室管理、排班、公告

**不包含范围**（可后续扩展）：住院管理、手术室管理、体检中心、医保实时结算接口。

### 1.4 术语定义

| 术语 | 英文 | 说明 |
|:----:|:----:|:----:|
| HIS | Hospital Information System | 医院信息系统 |
| EMR | Electronic Medical Record | 电子病历 |
| 号源 | Appointment Slot | 医生某一时段可接诊的病人数量 |
| 挂号 | Registration | 病人在医院登记就诊并获取排队序号 |
| 预约 | Appointment | 病人提前选择未来某日某医生就诊 |
| 处方 | Prescription | 医生为病人开具的用药/治疗指令 |
| 划价 | Pricing | 根据处方内容计算应收费用 |

## 2. 用户角色与权限体系

### 2.1 角色定义

系统采用 RBAC（基于角色的访问控制）模型，定义以下角色。其中**加粗**为当前系统已实现的四种核心角色，其余为 HIS 实际应包含的扩展角色。

| nurse | 护士 | 负责分诊、病人基本信息核对、医嘱执行、生命体征录入 | ⬜ 规划中 |
| pharmacist | 药房人员/药剂师 | 负责药品入库、处方审核、发药、库存盘点、报损 | ⬜ 规划中 |

| guide | 导医 | 负责病人引导、科室指引、就诊流程协助 | ⬜ 规划中 |
| super_admin | 超级管理员 | 负责系统运维、数据备份恢复、日志审计、权限分配 | ⬜ 规划中 |

## 3. 功能需求

### 3.1 登录认证模块（已实现）

#### 3.1.1 用户注册
- **需求描述**：新病人通过系统自助注册，注册时系统自动创建病人档案和登录账号。
- **输入**：用户名、密码、身份证号、地址、性别、手机号、生日
- **处理**：
  - 验证身份证号唯一性
  - 验证手机号格式
  - 病人信息存入 `hoimsystem_patient`，账号信息存入 `hoimsystem_users`
  - 角色自动设为 `patient`
- **输出**：注册成功/失败提示
- **约束**：身份证号为唯一标识，不可重复注册

#### 3.1.2 用户登录
- **需求描述**：已注册用户通过用户名和密码登录系统。
- **安全要求**：
  - 前端使用 RSA 公钥加密密码后传输
  - 后端校验用户名和密码
  - 登录成功后返回 accessToken 作为会话标识
- **输出**：accessToken、用户权限信息

#### 3.1.3 获取用户信息
- **需求描述**：根据 accessToken 获取当前登录用户的详细信息。
- **输出**：用户名、角色权限数组、头像地址

#### 3.1.4 用户登出
- **需求描述**：用户退出登录，清除会话状态。

---

### 3.2 病人管理模块

#### 3.2.1 病人信息录入（已实现：注册时自动录入）
- **需求描述**：首次就诊的病人在系统中建立档案。
- **字段**：姓名、性别、身份证号、生日、手机号、地址、过敏史
- **约束**：身份证号为唯一标识

#### 3.2.2 病人信息查询（已实现）
- **需求描述**：管理员可查询全部病人信息，病人可查询本人信息。
- **查询条件**：姓名、身份证号、手机号（支持模糊查询）
- **输出**：病人列表（含基本信息）

#### 3.2.3 病人信息维护（待实现）
- **需求描述**：修改病人档案信息。
- **约束**：身份证号不可修改；修改需记录操作日志

---

### 3.3 医生管理模块

#### 3.3.1 医生注册（已实现）
- **需求描述**：管理员或科室主任录入医生信息，同时创建医生登录账号。
- **输入**：登录用户名、密码、姓名、职称、性别、手机号、所属科室、权限、学历
- **处理**：
  - 校验用户名唯一性
  - 权限为 `director` 时，角色设为 `director`；否则设为 `doctor`
  - 创建 `hoimsystem_users` 和 `hoimsystem_doctor` 记录
- **输出**：注册成功/失败提示

#### 3.3.2 医生信息查询（已实现）
- **需求描述**：查询医生列表及详细信息。
- **输出**：医生 ID、姓名、性别、学历、手机号、权限、职称

#### 3.3.3 医生信息维护（待实现）
- **需求描述**：修改医生基本信息、调整所属科室、变更职称等。
- **约束**：涉及科室调动时需处理历史数据归属

#### 3.3.4 医生考勤签到（待实现）
- **需求描述**：医生每日上班签到，记录出勤状态。
- **业务规则**：签到时间与排班时间对比，计算迟到/早退/缺勤

### 3.4 科室管理模块（已实现）

#### 3.4.1 科室信息录入
- **需求描述**：管理员创建科室信息。
- **输入**：科室名称、电话、位置、主任医生 ID
- **约束**：科室名称唯一；主任医生必须是已注册医生

#### 3.4.2 科室信息查询
- **需求描述**：查询全部科室列表。
- **输出**：科室 ID、名称、电话、位置、主任姓名

#### 3.4.3 科室信息维护（待实现）
- **需求描述**：修改科室信息、更换科室主任。
- **约束**：更换主任时，旧主任权限需同步调整

---

### 3.5 排班管理模块（已实现）

#### 3.5.1 排班设置
- **需求描述**：为医生设置每周出诊排班。
- **输入**：医生 ID、出诊星期、出诊时段、号源数量、是否专家号
- **业务规则**：
  - 同一医生在同一星期同一时段只能有一条排班记录
  - 重复设置则更新原有排班
  - 号源数必须大于 0

#### 3.5.2 排班查询
- **需求描述**：查询医生的出诊排班信息。
- **权限控制**：
  - 管理员/病人：查看所有医生排班
  - 医生/主任：仅查看本人排班
- **输出**：医生 ID、姓名、排班数组、号源数、是否专家号

---

### 3.6 挂号管理模块（已实现）

#### 3.6.1 现场挂号
- **需求描述**：病人选择当日出诊医生进行现场挂号。
- **输入**：排班 ID、医生 ID、科室 ID、是否专家号
- **处理**：
  - 校验病人身份
  - 校验是否已挂同医生同类型的未就诊号
  - 号源数减 1
  - 生成挂号记录（状态：未就诊）
- **输出**：挂号成功/失败提示

#### 3.6.2 挂号记录查询
- **需求描述**：病人查询个人挂号记录，管理员查询全部。
- **输出**：挂号 UUID、序号、医生、是否专家号、科室、挂号时间、状态

#### 3.6.3 挂号取消
- **需求描述**：病人取消已挂号的记录。
- **处理**：将挂号状态设为"已取消"
- **约束**：已就诊的挂号不可取消

#### 3.6.4 排队叫号（待实现）
- **需求描述**：系统根据挂号时间自动生成排队序号，支持叫号、过号处理。
- **业务规则**：
  - 预约病人优先于现场挂号病人
  - 过号后自动顺延 5 个号
  - 叫号时显示当前排队人数

---

### 3.7 预约管理模块（已实现）

#### 3.7.1 预约挂号
- **需求描述**：病人提前选择未来某日某医生进行预约。
- **输入**：排班 ID、预约日期、医生 ID、科室 ID、偏好时段、是否专家号
- **处理**：
  - 校验病人身份
  - 校验预约日期是否在未来 7 天内且非当天
  - 号源数减 1
  - 生成预约记录（状态：未就诊）
- **输出**：预约成功/失败提示

#### 3.7.2 预约记录查询
- **需求描述**：查询个人预约记录。
- **输出**：预约 UUID、医生、是否专家号、科室、预约日期、偏好时段、状态

#### 3.7.3 预约取消
- **需求描述**：取消未就诊的预约。
- **处理**：将预约状态设为"已取消"
- **约束**：就诊前一天允许取消，就诊当天不可取消

#### 3.7.4 违约处理（待实现）
- **需求描述**：系统自动记录病人违约行为并执行处罚。
- **业务规则**：
  - 一个月内三次预约未取号，暂停预约资格一个月
  - 违约次数记录在 `hoimsystem_breach_record` 表中

### 3.8 病历管理模块（待实现）

#### 3.8.1 病历书写
- **需求描述**：医生为就诊病人书写电子病历。
- **输入**：就诊时间、医生 ID、病人 ID、症状描述、诊断结果
- **约束**：每个挂号/预约对应一条病历记录

#### 3.8.2 病历查询
- **需求描述**：按病人查询历史病历。
- **权限控制**：
  - 医生可查看本人接诊的病历
  - 病人可查看本人病历
  - 管理员可查看全部病历

### 3.9 检查检验管理模块（待实现）

#### 3.9.1 检查申请
- **需求描述**：医生为病人开具检查检验申请单。
- **字段**：检查类型、检查部位、检查目的、紧急程度

#### 3.9.2 检查结果录入
- **需求描述**：检验科室录入检查结果。
- **字段**：检查流水号、检查时间、检查结果（正常/异常）、异常描述

#### 3.9.3 检查结果查询
- **需求描述**：医生和病人查看检查结果。
- **输出**：检查名称、检查时间、结果、异常描述

---

### 3.10 处方管理模块（已实现）

#### 3.10.1 处方开具
- **需求描述**：医生为病人开具用药处方。
- **输入**：病人 ID、药品列表（药品 ID + 数量）
- **处理**：
  - 校验医生和病人信息
  - 校验药品库存充足
  - 扣减药品库存
  - 生成处方记录
  - 自动生成收费记录（状态：未缴费）
- **输出**：处方开具成功/失败提示

#### 3.10.2 处方查询
- **需求描述**：查询处方记录。
- **权限控制**：
  - 管理员：全部处方
  - 医生：本人开具的处方
  - 病人：本人的处方
- **输出**：处方 UUID、医生信息、病人信息、药品明细（名称+数量）

#### 3.10.3 处方修改与作废（待实现）
- **需求描述**：在取药前允许修改处方，取药后不可修改。
- **约束**：修改处方时需重新划价、调整库存

### 3.11 收费管理模块（已实现）

#### 3.11.1 收费记录查询
- **需求描述**：查询收费记录。
- **权限控制**：
  - 管理员：全部收费记录
  - 病人：本人的收费记录
- **输出**：收费 ID、创建时间、缴费时间、处方 ID、金额、状态

#### 3.11.2 在线缴费
- **需求描述**：病人对未缴费记录进行支付。
- **处理**：更新收费状态为"已缴费"，记录缴费时间
- **输出**：缴费成功/失败提示

#### 3.11.3 退费处理（待实现）
- **需求描述**：对已缴费但未取药的记录进行退费。
- **处理**：退费后恢复收费状态为"未缴费"或"已退费"

#### 3.11.4 发票管理（待实现）
- **需求描述**：缴费成功后生成电子发票。
- **字段**：发票号、收费项目、金额、开票时间

### 3.12 药品管理模块（已实现）

#### 3.12.1 药品入库
- **需求描述**：录入新采购的药品信息。
- **输入**：药品名称、库存、单价、过期日期、供应商、备注
- **处理**：自动生成采购时间
- **约束**：同一药品按批次分别入库（当前系统未实现批次管理）

#### 3.12.2 药品查询
- **需求描述**：查询药品库存信息。
- **输出**：药品 ID、名称、库存、单价、过期日期、采购时间、供应商、备注

#### 3.12.3 库存查询
- **需求描述**：按药品 ID 查询当前库存。
- **输出**：库存数量

#### 3.12.4 库存预警（待实现）
- **需求描述**：系统自动监控药品库存和有效期。
- **业务规则**：
  - 库存低于安全阈值时提醒采购
  - 药品过期前 1 个月提醒处理

#### 3.12.5 药品出库/发药（已实现：处方开具时自动出库）
- **需求描述**：处方确认时自动扣减药品库存。
- **约束**：库存不足时禁止开具处方

### 3.13 通知公告模块（已实现）

#### 3.13.1 发布公告
- **需求描述**：管理员或科室主任发布系统通知。
- **输入**：标题、内容、是否紧急、目标人群、过期时间
- **约束**：标题长度不超过 12 个字符

#### 3.13.2 公告查询
- **需求描述**：按角色查看可见的通知公告。
- **过滤规则**：
  - 主任：查看目标人群含"科室主任"的公告
  - 医生：查看目标人群含"医生"的公告
  - 病人：查看目标人群含"病人"的公告
- **输出**：UUID、标题、内容、是否紧急、目标人群、发送时间、过期时间、阅读数、发布人

## 5. 接口需求

### 5.1 内部接口

系统内部采用 RESTful API 风格，所有接口统一前缀为 `/api`，返回格式为：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

- `code = 200`：业务成功
- `code = 500`：业务失败，`msg` 返回错误信息

### 5.2 外部接口（待实现）

| 接口 | 说明 | 优先级 |
|:----:|:----:|:------:|
| 医保结算接口 | 与医保系统对接实现实时结算 | 高 |
| 短信接口 | 预约提醒、就诊提醒短信通知 | 中 |
| 微信/支付宝支付 | 在线缴费支付通道 | 中 |
| LIS/PACS 接口 | 检查检验系统数据交换 | 低 |

## 6. 非功能性需求

### 6.1 性能需求

| 指标 | 要求 |
|:----:|:----:|
| 页面响应时间 | 普通操作 ≤ 2s，复杂查询 ≤ 5s |
| 并发用户数 | 支持不少于 100 人同时在线 |
| 数据量 | 支持百万级历史数据存储 |

### 6.2 安全需求

- **传输安全**：密码使用 RSA 公钥加密传输
- **身份认证**：基于 accessToken 的会话认证
- **权限控制**：接口级别权限校验，防止越权访问
- **数据安全**：核心数据（病人隐私、处方）需脱敏展示
- **审计追踪**：关键操作记录日志，支持追溯

### 6.3 可靠性需求

- **事务一致性**：挂号、处方、收费等涉及多表操作必须使用数据库事务，失败时自动回滚
- **数据备份**：每日自动全量备份数据库
- **容错处理**：系统异常时返回统一错误格式，不暴露内部堆栈信息

### 6.4 可维护性需求

- **代码规范**：遵循 PEP8 编码规范
- **文档完整**：API 文档、数据库文档、需求文档同步更新
- **配置化管理**：系统参数通过配置文件管理，避免硬编码

### 6.5 兼容性需求

- **浏览器兼容**：支持 Chrome、Firefox、Edge、Safari 主流版本
- **移动端适配**：支持响应式布局，适配平板和手机访问
